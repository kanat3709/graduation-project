1. サービス概要
<旅行ログアプリ>
本サービスは、旅行の思い出を「場所」と「感情」で記録 / 振り返ることができる旅行ログアプリです。
写真だけでは曖昧になりがちな記憶を、位置情報と感情タグで補完し、体験そのものを残せるようにします。
忙しい社会人でも、簡単な操作で思い出を記録 / 振り返りできる仕組みを提供します。

2. このアイデアはどこから生まれたか
2-1. きっかけとなった体験・感情
2014年〜2015年にカナダへ留学していた際の写真を、帰国後しばらくしてから見返したとき、当時の記憶や感情が鮮明に蘇る体験をしました。

また、2025年に金沢旅行から帰宅後も、写真を見返しながら
「楽しかった」「また行きたい」といった感情や、その場のワクワク感を思い出しました。

一方で時間が経つにつれて、
・この写真はどこで撮ったのか
・そのとき何を感じていたのか
が曖昧になっていくことに気づきました。

本来、残したいはずの思い出が、写真の整理不足によって薄れていくことに対し、違和感や切なさを感じました。

2-2. なぜそれが「気になった」のか
この課題が気になった背景には、自分の性格や過去の経験があります。
・新しい場所や体験に価値を感じやすい（好奇心が強い）
・物事を整理して残しておきたい（几帳面な性格）

また過去に、
・写真データが埋もれて見つからなくなった経験
・その際に感じた強い喪失感や後悔
があり、「思い出を適切に残すこと」に対する意識が強くなっています。

自分にとって旅行は単なる観光ではなく、
「その場所で何を感じたか」という体験そのものに価値があるため、
それが失われていくことに対して強い違和感を持ちました。

3. 課題の整理
3-1. 表に見えている困りごと
・写真はあるが、どこで撮ったか分かりづらい
・時間が経つと記憶や感情が曖昧になる

なぜ？
・写真を整理できていない
→写真を整理する時間や余力がない
・感情や体験が紐づいていない
→整理するツールがない
・他人の旅行体験も見たいニーズがある

3-2. 本当に解決したい課題
なぜ困るのか？
・写真が整理されていない
→ 必要な情報にすぐアクセスできない
・なぜ？
→ 見返す機会が減る
・なぜ？
→ 思い出や感情が記録されないまま失われる

さらに、
・社会人は忙しく振り返る時間がない
・人とのつながりも減少する

＜本当に解決したい課題＞
・忙しい日常の中でも、思い出や感情を無理なく記録 / 振り返れる仕組みをつくる
・「思い出(写真)をきっかけに、他者とのゆるやかなつながりを生み出す」

3-3. 解決策のイメージ
例：金沢旅行
・写真（兼六園）
・感情タグ（感動、また行きたい）
・一言メモ
・地図に「金沢」のピン
→ 後から見返したときに、写真だけでなく「感情」まで蘇る

さらに、
・公開した写真は、共有画面(写真展のようなイメージ)で他のユーザーが閲覧できる
・同じ場所に行った人や、興味を持った人から「いいね」がもらえる
→ 自分の思い出が「誰かの共感」につながる
→ 思い出を残すこと自体が楽しくなり、継続につながる

8. 懸念点と対策
懸念点①
・問題：入力が面倒で使われない
・理由：記録系アプリは継続されにくい
・対策：
　- 入力項目を最小限に
　- 感情タグの選択式
　- UIの簡略化

懸念点②
・問題：プライバシーの懸念
・理由：位置情報や写真を公開することに抵抗がある人もいる
・対策：
　- 公開 / 非公開設定を実装
　- デフォルトは非公開にする
　- 位置情報の精度を調整できるようにする（例: 「金沢市」まで）

懸念点③
・問題：不適切な写真が投稿される可能性がある 
・理由：ユーザーが自由に写真を投稿できる仕組みの場合、公序良俗に反する内容や、他者が不快に感じるコンテンツが投稿されるリスクがあるため
・対策：
　- MVP段階では実装しない(ユーザー数が少ないため、リスクも限定的)
　- 将来的に実装を検討する機能：
    - 通報機能
    - 管理者による投稿の確認・削除対応
    - 利用規約の明示
    - AIによる画像判定や自動フィルタリングの導入

10. 技術スタック
10-1. 使用予定
・フレームワーク：Ruby on Rails 7.2
・言語：Ruby 3.3
・DB：PostgreSQL 16
・デプロイ：Render / Fly.io
・認証：Sorcery 0.18
・位置情報：Geocoder 1.x
・画像アップロード：Active Storage(Rails 7.2標準機能)

10-2. 技術選定理由
Ruby on Rails 7.2 を選んだ理由：
・RUNTEQのカリキュラムで学習中の技術であり、理解が深い
・MVPの開発に必要な機能が揃っている
・困ったときに調べられる日本語記事が豊富

PostgreSQL 16 を選んだ理由：
・Railsとの相性が良く、標準的に使われている
・位置情報を扱う際の拡張機能(PostGIS)も利用可能
・Render / Fly.io でのデプロイがスムーズ

Sorcery 0.18 を選んだ理由：
・RUNTEQのカリキュラムで学習済みの技術であり、理解が深い
・シンプルで柔軟性が高く、必要な機能を追加しやすい
・実務でも使われており、ポートフォリオとしても評価されやすい

Geocoder 1.x を選んだ理由：
・位置情報の取得 / 変換が簡単にできる
・Railsとの統合がスムーズ
・日本語の記事や事例が多い

Active Storage を選んだ理由：
・Railsに標準で組み込まれており、追加のgemが不要
・画像のアップロード / 表示が簡単に実装できる
・クラウドストレージ(AWS S3など)との連携も可能
